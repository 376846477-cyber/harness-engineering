# 分层架构设计

定义系统的逻辑层次结构，实现关注点分离。

## 常见分层模式

### 1. 三层架构

最经典的分层模式，适用于大多数业务系统。

```
┌─────────────────────────────────────────────────────────┐
│                    表示层（Presentation）                 │
│  • 用户界面                                              │
│  • API 网关                                              │
│  • 请求验证                                              │
│  • 响应格式化                                            │
├─────────────────────────────────────────────────────────┤
│                    业务层（Business）                     │
│  • 业务逻辑                                              │
│  • 领域服务                                              │
│  • 业务规则验证                                          │
│  • 事务管理                                              │
├─────────────────────────────────────────────────────────┤
│                    数据层（Data）                         │
│  • 数据访问                                              │
│  • 外部服务调用                                          │
│  • 缓存管理                                              │
│  • 消息队列                                              │
└─────────────────────────────────────────────────────────┘
```

**职责划分：**

| 层次 | 职责 | 不应该做的事 |
|------|------|-------------|
| 表示层 | 接收请求、返回响应、格式转换 | 业务逻辑、数据访问 |
| 业务层 | 业务逻辑、规则验证、流程编排 | 数据库操作、HTTP 处理 |
| 数据层 | 数据存储、外部服务集成 | 业务决策、请求处理 |

### 2. 整洁架构（Clean Architecture）

由 Robert C. Martin 提出，强调依赖规则。

```
┌───────────────────────────────────────────────────────────────┐
│                        外层：框架与驱动器                        │
│  ┌─────────────────────────────────────────────────────────┐  │
│  │                      接口适配器                           │  │
│  │  ┌───────────────────────────────────────────────────┐  │  │
│  │  │                    用例层                          │  │  │
│  │  │  ┌─────────────────────────────────────────────┐  │  │  │
│  │  │  │                  实体层                      │  │  │  │
│  │  │  │          核心业务规则和业务对象               │  │  │  │
│  │  │  └─────────────────────────────────────────────┘  │  │  │
│  │  │              应用特定业务规则                      │  │  │
│  │  └───────────────────────────────────────────────────┘  │  │
│  │              格式转换、控制器、网关                      │  │
│  └─────────────────────────────────────────────────────────┘  │
│              数据库、Web框架、外部服务                         │
└───────────────────────────────────────────────────────────────┘

依赖规则：源代码依赖只能指向内层
```

**各层职责：**

| 层次 | 包含内容 | 依赖方向 |
|------|----------|----------|
| 实体层 | 业务实体、核心业务规则 | 无外部依赖 |
| 用例层 | 应用业务规则、用例编排 | 依赖实体层 |
| 接口适配器 | 控制器、网关、转换器 | 依赖用例层 |
| 框架与驱动器 | 数据库、Web框架、外部服务 | 依赖适配器层 |

### 3. 六边形架构（端口与适配器）

强调应用核心与外部世界的隔离。

```
                        ┌─────────────┐
                        │  REST API   │
                        │  (适配器)    │
                        └──────┬──────┘
                               │
    ┌─────────────┐           │           ┌─────────────┐
    │   Web UI    │           │           │   消息队列   │
    │  (适配器)    │           │           │  (适配器)    │
    └──────┬──────┘           │           └──────┬──────┘
           │                  │                  │
           │    ┌─────────────┴─────────────┐    │
           └───→│                           │←───┘
                │        应用核心            │
                │        (六边形)            │
                │                           │
                │  • 领域模型               │
                │  • 业务服务               │
                │  • 端口定义               │
                │                           │
           ┌───→│                           │←───┐
           │    └─────────────┬─────────────┘    │
           │                  │                  │
    ┌──────┴──────┐           │           ┌──────┴──────┐
    │   数据库     │           │           │   邮件服务   │
    │  (适配器)    │           │           │  (适配器)    │
    └─────────────┘           │           └─────────────┘
                        ┌──────┴──────┐
                        │  第三方API  │
                        │  (适配器)    │
                        └─────────────┘
```

**核心概念：**

- **端口（Port）**：应用核心定义的接口
- **适配器（Adapter）**：端口的实现，连接外部世界
- **入站适配器**：接收外部请求（如 REST Controller）
- **出站适配器**：调用外部系统（如数据库 Repository）

### 4. 洋葱架构

类似整洁架构，强调层次化的依赖规则。

```
┌─────────────────────────────────────────────────────────────┐
│                         基础设施层                           │
│  ┌───────────────────────────────────────────────────────┐  │
│  │                      应用服务层                         │  │
│  │  ┌─────────────────────────────────────────────────┐  │  │
│  │  │                   领域服务层                      │  │  │
│  │  │  ┌───────────────────────────────────────────┐  │  │  │
│  │  │  │                领域模型层                  │  │  │  │
│  │  │  │         核心业务实体和值对象               │  │  │  │
│  │  │  └───────────────────────────────────────────┘  │  │  │
│  │  │            领域服务、规格                       │  │  │
│  │  └─────────────────────────────────────────────────┘  │  │
│  │              应用用例、事务管理                       │  │
│  └───────────────────────────────────────────────────────┘  │
│              外部服务、持久化、消息                          │
└─────────────────────────────────────────────────────────────┘
```

## 分层原则

### 1. 依赖规则

```
外层依赖内层，内层不知道外层的存在

表示层 ──依赖──→ 业务层 ──依赖──→ 数据层
   │               │               │
   │               │               │
   └── 不知道 ─────┘               │
                   └── 不知道 ─────┘
```

### 2. 数据流

```
请求：外层 → 内层
响应：内层 → 外层

用户请求 → 表示层 → 业务层 → 数据层
                ↓
用户响应 ← 表示层 ← 业务层 ← 数据层
```

### 3. 变化隔离

| 变化类型 | 影响范围 | 示例 |
|----------|----------|------|
| UI 变化 | 只影响表示层 | 更换前端框架 |
| 业务规则变化 | 只影响业务层 | 修改折扣计算规则 |
| 存储变化 | 只影响数据层 | 更换数据库 |

## 分层决策指南

```
                    ┌─────────────────┐
                    │ 需要分层架构吗？ │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ↓              ↓              ↓
         简单应用        中等复杂度       大型系统
         (单体脚本)      (业务系统)      (企业级)
              │              │              │
              ↓              ↓              ↓
         可能不需要      三层架构       整洁架构/
         或简单分层                    六边形架构
```

**选择建议：**

| 系统规模 | 推荐架构 | 理由 |
|----------|----------|------|
| 小型/原型 | 简单分层 | 快速开发，足够灵活 |
| 中型业务系统 | 三层架构 | 清晰职责，易于理解 |
| 大型企业系统 | 整洁架构/六边形 | 高度解耦，易于测试和替换 |
| 微服务 | 六边形架构 | 便于服务边界隔离 |

## 反模式

| 反模式 | 问题 | 解决方案 |
|--------|------|----------|
| 贫血模型 | 业务逻辑都在业务层，实体只是数据容器 | 将业务逻辑移到实体中 |
| 逻辑泄露 | 表示层包含业务逻辑 | 严格分层，业务逻辑只在业务层 |
| 过度分层 | 简单系统使用复杂架构 | 根据系统复杂度选择合适的架构 |
| 循环依赖 | 层与层之间相互依赖 | 使用依赖倒置，定义接口解耦 |
