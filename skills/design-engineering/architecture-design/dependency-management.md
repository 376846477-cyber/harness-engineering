# 依赖关系管理

管理模块间的依赖关系，确保系统结构清晰、易于维护。

## 依赖分类

根据依赖的性质和测试策略进行分类。

| 类型 | 描述 | 示例 | 测试策略 |
|------|------|------|----------|
| **进程内** | 纯计算、内存状态、无 I/O | 工具函数、领域逻辑 | 直接测试，无需适配器 |
| **本地可替换** | 有本地测试替身 | 数据库（PGLite）、文件系统 | 使用替身测试 |
| **远程但可控** | 自己的服务跨网络边界 | 微服务、内部 API | 端口+适配器模式 |
| **真正外部** | 第三方服务 | Stripe、Twilio、支付网关 | Mock 适配器 |

## 依赖原则

### 1. 依赖倒置原则（DIP）

高层模块不依赖低层模块，都依赖抽象。

```
错误方式：

┌─────────────┐
│   业务层    │
└──────┬──────┘
       │ 直接依赖
       ↓
┌─────────────┐
│   数据层    │
│  MySQL实现  │
└─────────────┘

问题：更换数据库需要修改业务层


正确方式：

┌─────────────┐
│   业务层    │
└──────┬──────┘
       │ 依赖抽象
       ↓
┌─────────────┐
│   接口      │
│  Repository │
└──────┬──────┘
       │ 实现
       ↓
┌─────────────┐
│   数据层    │
│  MySQL实现  │
└─────────────┘

优势：更换数据库只需新增实现
```

### 2. 接口隔离原则（ISP）

不应强迫客户依赖它们不使用的方法。

```
错误方式：

┌─────────────────────────────────────────┐
│              IOrderService              │
├─────────────────────────────────────────┤
│ + createOrder()                         │
│ + cancelOrder()                         │
│ + queryOrder()                          │
│ + exportOrders()           ← 管理后台专用│
│ + statistics()             ← 管理后台专用│
└─────────────────────────────────────────┘
         ↑                    ↑
         │                    │
    ┌────┴────┐          ┌────┴────┐
    │ 用户服务 │          │ 管理后台 │
    └─────────┘          └─────────┘

问题：用户服务依赖了不需要的方法


正确方式：

┌───────────────────┐    ┌───────────────────┐
│   IOrderService   │    │ IOrderManagement  │
├───────────────────┤    ├───────────────────┤
│ + createOrder()   │    │ + exportOrders()  │
│ + cancelOrder()   │    │ + statistics()    │
│ + queryOrder()    │    └───────────────────┘
└───────────────────┘             ↑
         ↑                        │
         │                  ┌─────┴─────┐
    ┌────┴────┐             │  管理后台  │
    │ 用户服务 │             └───────────┘
    └─────────┘

优势：每个客户只依赖需要的方法
```

### 3. 缝（Seam）原则

缝是可以在不修改代码的情况下改变行为的地方。

```
缝的使用规则：

一个适配器 = 假设的缝（可能过度设计）
两个适配器 = 真正的缝（生产 + 测试）

示例：

┌─────────────────────────────────────────────────────────┐
│                    订单服务                              │
│                         │                               │
│                         ↓                               │
│              ┌─────────────────────┐                    │
│              │  PaymentGateway     │ ← 缝（接口）        │
│              └─────────────────────┘                    │
│                    /           \                        │
│                   /             \                       │
│                  ↓               ↓                      │
│         ┌─────────────┐   ┌─────────────┐              │
│         │StripeAdapter│   │ MockAdapter │              │
│         │  (生产)     │   │  (测试)     │              │
│         └─────────────┘   └─────────────┘              │
└─────────────────────────────────────────────────────────┘

两个适配器证明了缝的必要性
```

### 依赖矩阵

```
           订单  支付  库存  商品  通知
    ┌─────────────────────────────────┐
订单 │  -    ×    ×    -    -        │
支付 │  -    -    -    -    ×        │

    └─────────────────────────────────┘

× = 依赖

```

## 循环依赖

### 循环依赖的危害

```
循环依赖：

┌──────────┐      ┌──────────┐
│  模块A   │ ───→ │  模块B   │
└────┬─────┘      └────┬─────┘
     ↑                 │
     └─────────────────┘

问题：
• 无法独立编译
• 无法独立测试
• 修改影响范围大
• 理解难度增加
```

### 解决循环依赖的方法

**方法1：提取公共模块**

```
原结构（循环依赖）：

┌──────────┐      ┌──────────┐
│  模块A   │ ←──→ │  模块B   │
└──────────┘      └──────────┘

解决方案：

┌──────────┐      ┌──────────┐
│  模块A   │      │  模块B   │
└────┬─────┘      └────┬─────┘
     │                 │
     └────────┬────────┘
              ↓
       ┌──────────┐
       │  公共模块 │
       └──────────┘
```

**方法2：依赖倒置**

```
原结构（循环依赖）：

┌──────────┐      ┌──────────┐
│  模块A   │ ←──→ │  模块B   │
└──────────┘      └──────────┘

解决方案：

┌──────────┐      ┌──────────┐
│  模块A   │ ───→ │  接口I   │ ←─── 模块B
└──────────┘      └──────────┘
                       ↑
                       │
                  ┌────┴────┐
                  │ 模块B实现│
                  └─────────┘
```

**方法3：事件驱动**

```
原结构（循环依赖）：

┌──────────┐      ┌──────────┐
│  模块A   │ ←──→ │  模块B   │
└──────────┘      └──────────┘

解决方案：

┌──────────┐             ┌──────────┐
│  模块A   │ ──发布事件─→ │  模块B   │
└──────────┘      ↓       └──────────┘
             ┌─────────┐
             │事件总线  │
             └─────────┘

模块A发布事件，模块B订阅事件
两者不直接依赖
```
