# 模块划分

将系统分解为高内聚、低耦合的模块，每个模块有清晰的职责和边界。

## 模块深度原则

来自《软件设计哲学》（John Ousterhout）。

### 深度模块 vs 浅层模块

**深度模块 = 小接口 + 大实现**

```
┌─────────────────────────────────────┐
│           小接口（2-3个方法）         │
├─────────────────────────────────────┤
│                                     │
│                                     │
│            大实现                    │
│        复杂逻辑隐藏在内部             │
│                                     │
│                                     │
└─────────────────────────────────────┘

优势：
• 调用者学习成本低
• 实现细节被隐藏
• 修改实现不影响调用者
• 测试只需验证接口行为
```

**浅层模块 = 大接口 + 小实现（应避免）**

```
┌─────────────────────────────────────┐
│              大接口（10+个方法）              │
├─────────────────────────────────────┤
│          小实现                              │
│      只是透传或简单包装                      │
└─────────────────────────────────────┘

问题：
• 调用者需要了解很多细节
• 没有隐藏复杂性
• 修改可能影响多个调用者
• 增加系统复杂度
```

### 删除测试

判断模块是否有价值的方法：

```
想象删除这个模块：

如果复杂性消失了 → 模块只是透传（浅层）
如果复杂性在 N 个调用者中重现 → 模块有价值（深层）
```

## 模块划分方法

### 1. 按业务能力划分

每个模块对应一个业务能力。

```
电商系统示例：

┌─────────────────────────────────────────────────────────┐
│                       电商系统                           │
├─────────────┬─────────────┬─────────────┬───────────────┤
│   商品模块   │   订单模块   │   用户模块   │   支付模块    │
├─────────────┼─────────────┼─────────────┼───────────────┤
│ • 商品管理  │ • 订单处理  │ • 用户注册  │ • 支付处理    │
│ • 分类管理  │ • 订单查询  │ • 用户认证  │ • 退款处理    │
│ • 库存管理  │ • 订单状态  │ • 权限管理  │ • 对账管理    │
└─────────────┴─────────────┴─────────────┴───────────────┘
```

### 2. 按子域划分（DDD）

根据领域驱动设计的子域划分模块。

```
┌─────────────────────────────────────────────────────────┐
│                       业务系统                           │
├─────────────────────────────────────────────────────────┤
│  核心域（Core Domain）                                   │
│  • 业务核心竞争力                                        │
│  • 投入最多资源                                          │
│  • 示例：电商的商品推荐、金融的风控模型                    │
├─────────────────────────────────────────────────────────┤
│  支撑域（Supporting Domain）                             │
│  • 支撑核心业务运作                                      │
│  • 可以外包或使用现成方案                                │
│  • 示例：支付、物流、通知                                │
├─────────────────────────────────────────────────────────┤
│  通用域（Generic Domain）                                │
│  • 所有业务都需要的                                      │
│  • 使用标准解决方案                                      │
│  • 示例：用户认证、权限管理、日志                        │
└─────────────────────────────────────────────────────────┘
```

### 3. 按变化原因划分（SRP）

单一职责原则：每个模块只有一个变化原因。

```
变化原因分析：

┌─────────────────┐     ┌─────────────────┐
│    订单模块      │     │    支付模块      │
├─────────────────┤     ├─────────────────┤
│ 变化原因：       │     │ 变化原因：       │
│ • 订单流程变更   │     │ • 支付渠道变更   │
│ • 订单规则变更   │     │ • 支付规则变更   │
└─────────────────┘     └─────────────────┘

如果两个变化原因经常一起变化 → 考虑合并
如果一个模块有多个变化原因 → 考虑拆分
```

## 模块接口设计

### 接口的定义

接口不仅是类型签名，还包括：

```
接口 = 类型签名 + 不变量 + 错误模式 + 性能特征 + 配置要求

示例：订单服务接口

类型签名：
  createOrder(items: CartItem[]): OrderId

不变量：
  • 订单创建后状态为 PENDING
  • 订单号唯一且有序
  • 订单金额 = 商品金额之和

错误模式：
  • 库存不足 → OutOfStockError
  • 商品下架 → ProductNotAvailableError
  • 系统繁忙 → RetryableError

性能特征：
  • 平均响应时间 < 500ms
  • 支持 1000 QPS

配置要求：
  • 需要配置库存服务地址
  • 需要配置订单号生成策略
```

### 接口设计原则

| 原则 | 描述 | 示例 |
|------|------|------|
| 最小化 | 接口方法越少越好 | 3-5 个核心方法 |
| 正交性 | 方法之间不重叠 | 不要 createOrder 和 createOrderAsync |
| 一致性 | 命名和风格统一 | 所有查询方法以 get/find 开头 |
| 可预测性 | 行为符合预期 | 相同输入产生相同输出 |

### 接口 vs 实现

```
┌─────────────────────────────────────────────────────────┐
│                       模块                               │
│  ┌───────────────────────────────────────────────────┐  │
│  │                    接口                           │  │
│  │  • 调用者需要知道的一切                           │  │
│  │  • 公开的类型和方法                               │  │
│  │  • 不变量和契约                                   │  │
│  └───────────────────────────────────────────────────┘  │
│                         │                               │
│                         ↓                               │
│  ┌───────────────────────────────────────────────────┐  │
│  │                    实现                           │  │
│  │  • 内部细节                                       │  │
│  │  • 可以自由修改                                   │  │
│  │  • 不影响调用者                                   │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

## 缝（Seam）与适配器

### 缝的定义

缝是可以在不修改代码的情况下改变行为的地方。

```
┌─────────────────────────────────────────────────────────┐
│                    调用者模块                            │
│                         │                               │
│                         ↓                               │
│              ┌─────────────────────┐                    │
│              │        缝           │ ← 行为可替换的点    │
│              │     (接口定义)       │                    │
│              └─────────────────────┘                    │
│                    /           \                        │
│                   /             \                       │
│                  ↓               ↓                      │
│         ┌─────────────┐   ┌─────────────┐              │
│         │   适配器A   │   │   适配器B   │              │
│         │  (生产实现)  │   │  (测试替身)  │              │
│         └─────────────┘   └─────────────┘              │
└─────────────────────────────────────────────────────────┘
```

### 适配器规则

```
一个适配器 = 假设的缝（可能过度设计）
两个适配器 = 真正的缝（生产 + 测试）

不要为只有一个实现的接口创建抽象
```

## 模块边界检查

### 检查清单

- [ ] 模块职责单一且明确
- [ ] 模块接口小而精
- [ ] 模块实现细节被隐藏
- [ ] 模块间依赖单向
- [ ] 模块可独立测试
- [ ] 模块可独立部署（如需要）

### 常见问题

| 问题 | 症状 | 解决方案 |
|------|------|----------|
| 模块过大 | 修改频繁，影响范围大 | 按职责拆分 |
| 模块过小 | 模块数量爆炸，调用链长 | 合并相关模块 |
| 模块耦合 | 修改一个模块影响其他 | 定义清晰接口 |
| 接口膨胀 | 接口方法越来越多 | 重新审视职责，拆分接口 |
