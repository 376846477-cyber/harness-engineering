# Clean Code总览 - 详细版

基于Clean Code指导书（IPD_MPD_SWD_W375913）

## 概述

Clean Code是指在满足功能正确的前提下，把具有以下**七个特征**的代码称为Clean Code：

| 特征 | 定义 | 优先级 |
|-----|------|-------|
| **可读** | 易于阅读 | 高 |
| **可维护** | 可修改、可扩展、可复用 | 高 |
| **安全** | 防御恶意威胁 | 高 |
| **可靠** | 正常运行概率 | 中 |
| **可测试** | 易于发现和定位故障 | 中 |
| **高效** | 资源利用率高 | 中 |
| **可移植** | 跨平台运行 | 低 |

## 1. 可读（Readability）

**定义**：代码可读是指易于阅读的能力。

**核心思想**：提升可读性就是要传递足够多的有用信息、减少信息失真（误解）、减少信息干扰、降低信息理解难度。

**关键实践**：
- 利用名称传递信息（好名字自注释）
- 利用格式传递信息（缩进、空行、风格一致）
- 注释作为代码的补充（不要给烂代码加注释）
- 减少嵌套层级（使用保护语句、早返回）
- 变量作用域尽量小

## 2. 可维护（Maintainability）

**定义**：可维护是指的软件可修改、可扩展和可复用的能力。

**核心思想**：修改/扩展代码要影响局部化，通过**抽象**带来的相对稳定的间接层来隔离变化。

**关键实践**：
- 通过抽象隔离变化（接口、泛型、多态）
- 缩小依赖范围（接口最少化、参数最少化）
- 向着稳定的方向依赖（依赖抽象而非实现）
- 引入中间层降低耦合（适配器、观察者模式）
- 抽取公共部分（DRY原则）

## 3. 安全（Security）

**定义**：代码的安全性是指软件对恶意威胁（如未授权访问/使用、泄露、破坏、篡改、毁灭）的防护能力。

**核心思想**：编码过程中开发人员要有攻击者思维，假设所有外部输入都是不可信的。

**关键实践**：
- 减少代码的攻击面
- 输入校验及转义（SQL注入、命令注入防护）
- 采用防御性的编程策略
- 敏感数据保护

**详细规范**：
- 输入安全：使用 `java-security-input`
- 加密安全：使用 `java-security-crypto`
- 序列化安全：使用 `java-security-serialization`

## 4. 可靠（Reliability）

**定义**：代码可靠性是软件在给定时间间隔和环境条件下，按设计要求成功运行程序的概率。

**核心思想**：系统必须预备预防错误、容错、故障恢复（自愈）的能力。

**关键实践**：
- 操作防呆设计
- 系统过载保护（负载均衡、流控）
- 消息重复发送（通信可靠性）
- 状态定时同步
- 防御式编程
- 资源管理（获取→使用→释放）
- 核心流程依赖最小化
- 降级处理策略

## 5. 可测试（Testability）

**定义**：代码可测试性是指软件发现故障并隔离、定位故障的能力。

**核心思想**：通过分治法，把复杂系统隔离测试，降低测试复杂度。

**关键实践**：
- 高内聚低耦合（便于隔离测试）
- 减少系统状态和全局变量
- 支持依赖注入
- 输入参数合理（不过多）
- 结果和过程可观测
- 异常日志适当记录

## 6. 高效（Performance）

**定义**：高效指系统资源利用率高，系统资源包括CPU、内存、硬盘、网络带宽等。

**核心思想**：在不影响其他代码质量属性的前提下，养成编写高效代码的习惯，不要过早做性能优化。

**关键实践**：
- 选择高效的算法（Hash、二分、红黑树）
- 减少冗余或无效计算（按需初始化）
- 合理使用并行编程
- 减少内存浪费（合理选择数据结构）
- 减少内存碎片（对象复用）
- IO优化（批量操作、缓冲IO）
- 数据库优化（JOIN、批量查询）

## 7. 可移植（Portability）

**定义**：可移植是指代码在不同硬件、操作系统、编译器环境下运行的能力。

**核心思想**：Java本身平台无关，但编码时需避免引入平台相关的依赖。

**关键实践**：
- 避免平台相关的API（路径分隔符、行分隔符）
- 使用标准API和跨平台库
- 明确指定字符集（UTF-8）
- 时区和locale处理
- JVM版本兼容性

## 特征权衡

在实际开发中，七个特征之间可能存在冲突，需要根据具体场景权衡：

- **可读 vs 高效**：过度优化可能损害可读性
- **可维护 vs 高效**：过度抽象可能影响性能
- **安全 vs 可移植**：某些安全特性可能依赖特定平台

建议优先级：**功能正确 > 可读 > 可维护 > 安全 > 可靠 > 可测试 > 高效 > 可移植**

## 完整检查清单

代码审查时可使用以下检查清单：

- [ ] 代码是否易于阅读和理解？
- [ ] 修改是否影响局部化（最小化影响范围）？
- [ ] 是否有安全漏洞（注入、加密等）？
- [ ] 是否有容错和自愈机制？
- [ ] 代码是否易于测试？
- [ ] 是否有不必要的性能浪费？
- [ ] 是否依赖平台特定的API？
