# C++ Clean Code总览

基于Clean Code原则与现代C++（C++11/14/17/20）最佳实践

## 概述

Clean Code是指在满足功能正确的前提下，把具有以下**七个核心特征**的代码称为Clean Code。这些特征相互关联、相互影响，共同构成高质量C++代码的基础。

| 特征 | 定义 | 优先级 | 关键指标 |
|-----|------|-------|---------|
| **可读** | 易于阅读理解 | 高 | 命名清晰、格式规范、注释得当 |
| **可维护** | 可修改、可扩展、可复用 | 高 | 低耦合、高内聚、模块化 |
| **安全** | 防御恶意威胁 | 高 | 无漏洞、数据保护、权限控制 |
| **可靠** | 正常运行概率 | 中 | 容错、恢复、防御式编程 |
| **可测试** | 易于发现和定位故障 | 中 | 可隔离、可观测、可模拟 |
| **高效** | 资源利用率高 | 中 | 算法优化、内存管理、并发 |
| **可移植** | 跨平台运行 | 低 | 标准库、抽象平台差异 |

## 特征优先级与权衡

### 优先级原则

1. **第一优先级**：可读、可维护、安全 — 贯穿整个开发周期
2. **第二优先级**：可靠、可测试、高效 — 视项目需求调整
3. **第三优先级**：可移植 — 视目标平台决定

### 权衡决策

- **可读 vs 高效**：优先可读，性能热点再优化
- **安全 vs 高效**：安全优先，禁止为性能牺牲安全
- **可测试 vs 高效**：通过设计模式实现兼顾
- **可移植 vs 高效**：使用条件编译隔离平台代码

## 核心特征详解

### 1. 可读性（Readability）

**定义**：代码应易于阅读和理解，降低认知负担。

**关键实践**：
- 利用名称传递信息：使用描述性命名，避免缩写和魔法数
- 利用格式传递信息：合理缩进、空行、代码分组
- 注释作为代码的补充：解释"为什么"而非"是什么"
- 减少嵌套层级：使用早返回、guard clause
- 使用现代C++惯用法：RAII、智能指针、范围for、auto
- 变量作用域尽量小：就近声明、初始化

**检查清单**：
- [ ] 变量和函数名称是否表达意图？
- [ ] 代码格式是否遵循团队规范？
- [ ] 是否存在过深的嵌套（>3层）？
- [ ] 是否合理使用auto避免冗余类型声明？
- [ ] 注释是否解释了复杂逻辑的原因？

**相关Skill**：`cpp-naming`、`cpp-formatting`、`cpp-comments`、`cpp-readability`

### 2. 可维护性（Maintainability）

**定义**：代码应易于修改、扩展和复用，适应需求变化。

**关键实践**：
- 通过抽象隔离变化：虚函数、模板、概念（C++20）
- 缩小依赖范围：前向声明、接口最少化、PIMPL模式
- 向着稳定的方向依赖：依赖抽象而非具体实现
- 引入中间层降低耦合：桥接模式、适配器模式
- 抽取公共部分：DRY原则、函数抽取、模板化

**检查清单**：
- [ ] 类是否单一职责？
- [ ] 依赖是否通过抽象接口？
- [ ] 是否遵循SOLID原则？
- [ ] 公共代码是否被抽取复用？
- [ ] 模块边界是否清晰？

**相关Skill**：`cpp-maintainability`

### 3. 安全性（Security）

**定义**：代码应能防御恶意威胁，保护系统和数据安全。

**关键实践**：
- 减少攻击面：最小权限原则、禁用不安全函数
- 输入校验及转义：所有外部输入都是不可信的
- 敏感数据保护：加密存储、安全传输、及时清零
- 防御性编程：边界检查、空指针检查、类型安全
- 禁止缓冲区溢出：使用std::array、std::vector、std::string

**检查清单**：
- [ ] 所有外部输入是否经过校验？
- [ ] 是否存在缓冲区溢出风险？
- [ ] 敏感数据是否加密存储？
- [ ] 是否使用安全的字符串处理函数？
- [ ] 密钥是否硬编码在代码中？

**相关Skill**：`cpp-security-input`、`cpp-security-crypto`、`cpp-security-serialization`

### 4. 可靠性（Reliability）

**定义**：代码应稳定运行，具备容错和恢复能力。

**关键实践**：
- RAII资源管理：自动释放、异常安全
- 操作防呆设计：状态机、输入验证、操作确认
- 系统过载保护：限流、熔断、降级
- 防御式编程：前置条件、后置条件、断言
- 降级处理策略：默认值、重试、回滚

**检查清单**：
- [ ] 资源是否使用RAII管理？
- [ ] 异常是否被正确处理？
- [ ] 是否有内存泄漏风险？
- [ ] 关键操作是否有容错机制？
- [ ] 是否使用断言验证不变量？

**相关Skill**：`cpp-reliability`、`cpp-exceptions`

### 5. 可测试性（Testability）

**定义**：代码应易于编写测试，快速发现和定位问题。

**关键实践**：
- 高内聚低耦合：单一职责、依赖注入
- 依赖注入：构造函数注入、接口注入
- 结果和过程可观测：日志、指标、追踪
- 使用GTest/GMock：单元测试、模拟对象

**检查清单**：
- [ ] 是否使用依赖注入便于mock？
- [ ] 函数是否有明确的输入输出？
- [ ] 全局状态是否可控？
- [ ] 是否有单元测试覆盖？
- [ ] 测试是否可重复执行？

**相关Skill**：`cpp-testability`

### 6. 高效性（Performance）

**定义**：代码应高效利用计算资源，满足性能需求。

**关键实践**：
- 选择高效的算法和数据结构：时间复杂度、空间复杂度
- 移动语义避免拷贝：std::move、右值引用
- 减少冗余或无效计算：延迟计算、缓存
- 内存管理优化：对象池、自定义分配器
- 零拷贝和对象池：避免不必要的拷贝

**检查清单**：
- [ ] 是否选择了正确的算法？
- [ ] 是否避免了不必要的拷贝？
- [ ] 是否合理使用移动语义？
- [ ] 是否存在性能热点？
- [ ] 内存分配是否优化？

**相关Skill**：`cpp-performance`

### 7. 可移植性（Portability）

**定义**：代码应能在不同平台上编译和运行。

**关键实践**：
- 使用std::filesystem处理路径：跨平台路径操作
- 明确字符编码：UTF-8、std::u8string
- 使用标准库和跨平台库：避免平台特定API
- 编译器兼容性：注意编译器扩展差异
- 字节序处理：网络字节序、htonl/ntohl

**检查清单**：
- [ ] 是否使用标准库替代平台API？
- [ ] 路径处理是否跨平台？
- [ ] 字符编码是否明确？
- [ ] 是否使用条件编译隔离平台代码？
- [ ] 是否考虑字节序问题？

**相关Skill**：`cpp-portability`

## 现代C++最佳实践

### 资源管理
- 使用智能指针（unique_ptr、shared_ptr）替代裸指针
- 遵循RAII原则，资源获取即初始化
- 使用std::vector、std::string替代C风格数组

### 类型安全
- 使用强类型枚举（enum class）
- 使用constexpr替代宏常量
- 使用using替代typedef

### 并发安全
- 使用std::atomic、std::mutex保护共享数据
- 使用std::lock_guard、std::unique_lock管理锁
- 优先使用无锁数据结构

### 错误处理
- 使用异常处理错误情况
- 使用std::optional表示可能失败的返回值
- 使用std::expected（C++23）或第三方库

## 代码审查检查清单总览

在代码审查时，应综合考虑七个特征：

1. **可读性**：命名、格式、注释是否规范？
2. **可维护性**：设计是否灵活、耦合是否合理？
3. **安全性**：是否存在安全隐患？
4. **可靠性**：错误处理是否完善？
5. **可测试性**：是否易于测试？
6. **高效性**：性能是否满足需求？
7. **可移植性**：是否需要跨平台支持？

根据项目特点，选择重点关注的特征，但前三个特征（可读、可维护、安全）应始终作为审查重点。
