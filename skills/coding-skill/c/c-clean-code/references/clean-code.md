# Clean Code总览 - C语言编码规范指南

基于Clean Code指导书，为C语言项目提供系统化的代码质量标准。

## 概述

Clean Code是指在满足功能正确的前提下，把具有以下**七个核心特征**的代码称为Clean Code。这些特征相互关联，共同构成高质量代码的基础。

### 七个核心特征总览

| 特征 | 定义 | 优先级 | 核心价值 |
|-----|------|-------|---------|
| **可读** | 易于阅读理解 | 高 | 降低认知负担，加速团队协作 |
| **可维护** | 可修改、可扩展、可复用 | 高 | 降低长期维护成本 |
| **安全** | 防御恶意威胁 | 高 | 保护系统和用户数据 |
| **可靠** | 正常运行概率高 | 中 | 提升系统稳定性 |
| **可测试** | 易于发现和定位故障 | 中 | 加速问题诊断和修复 |
| **高效** | 资源利用率高 | 中 | 优化运行效率 |
| **可移植** | 跨平台运行 | 低 | 扩大适用范围 |

### 特征优先级与权衡

**高优先级特征（必须满足）**：
- 可读性、可维护性、安全性是基础要求，任何情况下不可妥协

**中优先级特征（平衡考虑）**：
- 可靠性、可测试性、高效性需要根据项目实际需求权衡

**低优先级特征（按需满足）**：
- 可移植性取决于目标平台范围

**权衡原则**：
1. 可读性优先于效率（除非有明确的性能瓶颈）
2. 安全性不可为效率让步
3. 可测试性投入应在可维护性收益中体现
4. 可移植性投入应与目标平台数量匹配

## 1. 可读（Readability）

### 定义
代码应当像文档一样清晰，让读者能够快速理解代码的意图和实现逻辑。

### 核心原则
- **名称即文档**：通过命名传递意图，减少对注释的依赖
- **格式即结构**：通过排版展示代码的逻辑层次
- **注释补充说明**：解释"为什么"而非"是什么"

### 关键实践

| 实践 | 说明 | 示例 |
|-----|------|------|
| 有意义的命名 | 变量、函数名传达意图 | `timeout_ms` 而非 `t` |
| 一致的格式 | 遵循团队代码风格 | 统一的缩进、空行规则 |
| 适度注释 | 在必要处添加注释 | 解释复杂算法、业务规则 |
| 减少嵌套 | 避免深层嵌套结构 | 使用早返回、提取函数 |
| 小作用域 | 变量定义靠近使用处 | 循环变量在for中定义 |
| 单一职责 | 函数只做一件事 | 每个函数有明确的目标 |

### 检查清单
- [ ] 所有变量、函数名是否清晰表达意图？
- [ ] 代码格式是否符合团队规范？
- [ ] 复杂逻辑是否有注释说明？
- [ ] 嵌套层级是否超过3层？
- [ ] 变量作用域是否最小化？
- [ ] 函数长度是否合理（建议<50行）？

### 相关Skill
- `c-readability` - 可读性详细规范
- `c-naming` - 命名规范
- `c-formatting` - 排版格式规范
- `c-comments` - 注释规范

## 2. 可维护（Maintainability）

### 定义
代码应当易于修改、扩展和复用，降低长期维护成本。

### 核心原则
- **模块化设计**：高内聚低耦合
- **依赖管理**：向稳定方向依赖
- **DRY原则**：Don't Repeat Yourself

### 关键实践

| 实践 | 说明 | 示例 |
|-----|------|------|
| 头文件接口隔离 | 通过头文件暴露最小接口 | 隐藏实现细节 |
| 缩小依赖范围 | 减少模块间耦合 | 使用依赖注入 |
| 稳定依赖方向 | 依赖稳定的抽象层 | 依赖接口而非实现 |
| 模块化设计 | 功能边界清晰 | 单一职责模块 |
| 代码复用 | 抽取公共逻辑 | 提取公共函数库 |

### 检查清单
- [ ] 模块职责是否单一清晰？
- [ ] 头文件是否只暴露必要接口？
- [ ] 是否存在重复代码可抽取？
- [ ] 依赖关系是否清晰可控？
- [ ] 修改一处是否影响多处？
- [ ] 是否便于添加新功能？

### 相关Skill
- `c-maintainability` - 可维护性详细规范
- `c-error-handling` - 错误处理规范

---

## 3. 安全（Security）

### 定义
代码应当主动防御恶意威胁，保护系统和用户数据安全。

### 核心原则
- **最小攻击面**：减少可被利用的入口
- **零信任原则**：不信任任何外部输入
- **纵深防御**：多层安全机制

### 关键实践

| 实践 | 说明 | 示例 |
|-----|------|------|
| 减少攻击面 | 最小化暴露的接口和功能 | 关闭不必要端口 |
| 输入校验转义 | 所有外部输入不可信 | 边界检查、格式验证 |
| 敏感数据保护 | 加密存储传输 | 密码哈希、密钥管理 |
| 防御性编程 | 假设最坏情况 | 参数校验、错误处理 |
| 缓冲区安全 | 防止溢出攻击 | 使用安全函数、边界检查 |

### 检查清单
- [ ] 所有外部输入是否经过校验？
- [ ] 敏感数据是否加密存储？
- [ ] 是否使用安全的字符串/内存操作函数？
- [ ] 是否存在缓冲区溢出风险？
- [ ] 错误信息是否泄露敏感信息？
- [ ] 是否有权限控制和认证机制？

### 相关Skill
- `c-security-input` - 输入校验与安全规范
- `c-security-crypto` - 加密安全规范
- `c-security-serialization` - 序列化安全规范

## 4. 可靠（Reliability）

### 定义
代码应当稳定运行，正确处理异常情况，避免崩溃和数据损坏。

### 核心原则
- **资源管理**：每种资源都有明确的生命周期管理
- **防御设计**：预防常见错误模式
- **故障恢复**：设计降级和恢复机制

### 关键实践

| 实践 | 说明 | 示例 |
|-----|------|------|
| 内存管理配对 | 每个malloc有对应free | 使用RAII模式、统一清理 |
| 操作防呆设计 | 防止误操作 | 参数校验、状态检查 |
| 过载保护 | 防止资源耗尽 | 限流、队列管理 |
| 防御式编程 | 预期并处理错误 | 错误码、异常处理 |
| 降级策略 | 故障时优雅降级 | 默认值、重试机制 |

### 检查清单
- [ ] 动态内存是否正确释放？
- [ ] 文件句柄、网络连接是否正确关闭？
- [ ] 是否处理了所有可能的错误情况？
- [ ] 是否有超时和重试机制？
- [ ] 是否有资源耗尽的保护措施？
- [ ] 错误处理是否完整统一？

### 相关Skill
- `c-reliability` - 可靠性详细规范
- `c-error-handling` - 错误处理规范

---

## 5. 可测试（Testability）

### 定义
代码应当易于编写测试，便于发现和定位问题。

### 核心原则
- **可观测性**：内部状态可被检查
- **可隔离性**：依赖可被模拟替换
- **可重复性**：测试结果稳定可重复

### 关键实践

| 实践 | 说明 | 示例 |
|-----|------|------|
| 高内聚低耦合 | 模块独立可测 | 单一职责、依赖注入 |
| 函数指针注入 | 实现依赖替换 | 回调函数、接口抽象 |
| 结果可观测 | 提供状态查询接口 | getter函数、状态输出 |
| 单元测试框架 | 使用成熟测试工具 | Unity、CMock框架 |

### 检查清单
- [ ] 函数是否可以独立测试？
- [ ] 外部依赖是否可以模拟？
- [ ] 是否有清晰的测试边界？
- [ ] 测试覆盖是否全面？
- [ ] 是否有单元测试框架支持？
- [ ] 测试用例是否易于维护？

### 相关Skill
- `c-testability` - 可测试性详细规范

## 6. 高效（Performance）

### 定义
代码应当合理利用计算资源，在满足功能前提下追求最优效率。

### 核心原则
- **正确性优先**：先保证正确，再优化性能
- **测量驱动**：基于实际测量优化，避免过早优化
- **关注瓶颈**：优化真正的性能热点

### 关键实践

| 实践 | 说明 | 示例 |
|-----|------|------|
| 算法选择 | 选择合适的数据结构和算法 | 时间复杂度、空间复杂度 |
| 减少冗余计算 | 缓存计算结果 | 循环外提、结果缓存 |
| 内存优化 | 减少内存分配和拷贝 | 内存池、零拷贝 |
| 系统调用优化 | 减少不必要的系统调用 | 批量操作、缓冲区 |

### 检查清单
- [ ] 是否选择了合适的算法和数据结构？
- [ ] 是否存在明显的性能瓶颈？
- [ ] 是否避免了不必要的内存拷贝？
- [ ] 循环中是否有可外提的计算？
- [ ] 是否有性能测试验证优化效果？
- [ ] 内存使用是否高效合理？

### 相关Skill
- `c-performance` - 高效性详细规范

---

## 7. 可移植（Portability）

### 定义
代码应当能够适应不同平台环境，减少平台相关的修改。

### 核心原则
- **标准优先**：优先使用标准库和标准接口
- **抽象平台差异**：封装平台相关代码
- **明确假设**：文档化平台依赖

### 关键实践

| 实践 | 说明 | 示例 |
|-----|------|------|
| 标准库优先 | 使用标准C库函数 | 避免平台特定扩展 |
| 字符编码明确 | 统一字符编码处理 | UTF-8编码 |
| 跨平台构建 | 使用可移植构建系统 | CMake构建 |
| 条件编译 | 处理平台差异 | 预处理宏隔离 |
| 字节序处理 | 网络传输统一字节序 | htonl/ntohl转换 |

### 检查清单
- [ ] 是否使用了平台特定的API？
- [ ] 是否正确处理了字节序问题？
- [ ] 是否明确指定了字符编码？
- [ ] 路径处理是否跨平台兼容？
- [ ] 是否使用了跨平台构建系统？
- [ ] 平台差异是否通过宏隔离？

### 相关Skill
- `c-portability` - 可移植性详细规范

## 实践案例：综合应用示例

### 案例：设计一个安全的配置文件解析器

**需求**：读取并解析配置文件，返回键值对结构。

**应用七个特征**：

1. **可读**：函数命名`parse_config_file`，变量`config_entry`
2. **可维护**：模块化设计，接口在头文件定义
3. **安全**：输入校验、缓冲区边界检查、敏感信息保护
4. **可靠**：错误处理完整、内存正确释放
5. **可测试**：依赖注入文件读取接口，便于mock
6. **高效**：一次性读取文件，避免频繁IO
7. **可移植**：使用标准C库，跨平台路径处理

```c
// 良好实践示例
typedef struct {
    char key[MAX_KEY_LEN];
    char value[MAX_VALUE_LEN];
} config_entry_t;

int parse_config_file(const char* filepath,
                      config_entry_t* entries,
                      size_t max_entries,
                      size_t* out_count);
```

## 快速诊断指南

当发现代码质量问题时，按以下路径定位：

| 问题现象 | 可能特征 | 推荐Skill |
|---------|---------|----------|
| 难以理解代码意图 | 可读性 | `c-readability`, `c-naming` |
| 修改一处影响多处 | 可维护性 | `c-maintainability` |
| 存在安全漏洞 | 安全性 | `c-security-*` 系列 |
| 程序崩溃或内存泄漏 | 可靠性 | `c-reliability`, `c-error-handling` |
| 难以编写单元测试 | 可测试性 | `c-testability` |
| 性能不足 | 高效性 | `c-performance` |
| 无法在其他平台编译 | 可移植性 | `c-portability` |
