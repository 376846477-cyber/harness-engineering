# C语言注释规范详细参考

## 一、注释类型与格式

### 1.1 文件头注释
文件头注释应位于文件顶部，说明文件的基本信息：

```c
/**
 * @file    filename.c
 * @brief   文件功能简要描述
 * @author  作者姓名
 * @date    2024-01-01
 * @version 1.0.0
 * @copyright Copyright (c) 2024 公司名称
 * 
 * @details
 * 文件功能的详细描述，包括主要功能模块、设计思路等。
 */
```

### 1.2 函数注释
函数注释应位于函数定义之前，详细说明函数的功能、参数、返回值等：

```c
/**
 * @brief 计算两个整数的和
 * @param[in]  a 第一个加数
 * @param[in]  b 第二个加数
 * @return 两数之和
 * @retval 0    参数无效
 * @note 输入参数必须为正整数
 * @warning 不支持负数输入
 */
int add(int a, int b);
```

### 1.3 代码块注释
用于解释复杂算法或业务逻辑：

```c
/* 采用二分查找算法提高搜索效率，时间复杂度O(log n)
 * 适用于已排序的数组，对于小数组建议使用线性查找
 */
int binary_search(int *arr, int size, int target) {
    // 实现代码
}
```

### 1.4 行内注释
短小精悍，位于代码行末尾：

```c
int result = a + b;  // 计算两数之和
```

## 二、Doxygen标准格式

### 2.1 常用标签说明

| 标签 | 用途 | 示例 |
|------|------|------|
| @file | 文件名 | @file main.c |
| @brief | 简要描述 | @brief 初始化系统 |
| @param | 参数说明 | @param[in] size 数组大小 |
| @return | 返回值说明 | @return 成功返回0 |
| @retval | 特定返回值 | @retval -1 失败 |
| @note | 注意事项 | @note 线程不安全 |
| @warning | 警告信息 | @warning 可能内存泄露 |
| @see | 交叉引用 | @see init_system() |
| @deprecated | 弃用标记 | @deprecated 请使用new_func() |

### 2.2 参数方向标记

| 标记 | 说明 |
|------|------|
| [in] | 输入参数 |
| [out] | 输出参数 |
| [in,out] | 输入输出参数 |

## 三、正确与错误示例

### 3.1 函数注释示例

**正确示例：**
```c
/**
 * @brief 初始化网络连接
 * @param[in]  host 服务器地址，如"192.168.1.1"
 * @param[in]  port 端口号，范围1-65535
 * @param[out] fd  返回的socket文件描述符
 * @return 成功返回0，失败返回错误码
 * @retval -1 参数无效
 * @retval -2 连接超时
 * @retval -3 DNS解析失败
 * @note 此函数为阻塞调用，超时时间默认5秒
 * @see close_connection()
 */
int init_connection(const char *host, int port, int *fd);
```

**错误示例：**
```c
/* 初始化连接 */
int init_connection(const char *host, int port, int *fd);
```

### 3.2 代码注释示例

**正确示例：**
```c
/* 使用缓存减少数据库查询次数，提升性能约30%
 * 缓存有效期为60秒，过期后自动重新加载
 */
cache_t *cache = create_cache(60);

if (cache == NULL) {
    return ERR_CACHE_CREATE;  // 缓存创建失败，返回错误码
}
```

**错误示例：**
```c
// 创建缓存
cache_t *cache = create_cache(60);

// 判断是否为空
if (cache == NULL) {
    // 返回错误
    return -1;
}
```

## 四、注释原则

### 4.1 基本原则
1. **解释"为什么"而非"是什么"**：代码应自解释，注释应说明设计意图
2. **使用中文注释**：便于团队理解和维护
3. **保持同步更新**：修改代码时同步更新注释
4. **避免废话注释**：删除无意义的注释
5. **删除废弃注释**：注释掉的代码应删除，不保留

### 4.2 TODO/FIXME注释规范

```c
/* TODO(zhangsan): 添加参数校验，预计2024-02完成 */
/* FIXME(lisi): 存在内存泄露问题，需要修复 */
/* HACK(wangwu): 临时方案，等待上游API更新后重构 */
/* NOTE: 此处算法复杂度较高，后续考虑优化 */
```

格式：`/* 关键字(负责人): 描述 */`

### 4.3 注释位置规范
- 文件头注释：文件第一行
- 函数注释：函数定义前一行
- 代码块注释：代码块上一行
- 行内注释：代码右侧，用空格分隔

## 五、完整检查清单

### 5.1 文件头检查
- [ ] 包含@brief简要描述
- [ ] 包含作者和日期信息
- [ ] 版权声明完整（如需要）

### 5.2 函数注释检查
- [ ] 所有公共函数都有注释
- [ ] @brief描述清晰准确
- [ ] 参数说明包含方向标记[in]/[out]
- [ ] 返回值说明完整
- [ ] 特殊情况有@note/@warning说明

### 5.3 代码注释检查
- [ ] 复杂算法有解释说明
- [ ] 注释解释设计意图而非代码本身
- [ ] 没有废话注释和废弃注释
- [ ] 注释与代码保持同步

### 5.4 格式规范检查
- [ ] 使用Doxygen标准格式
- [ ] 注释语言统一使用中文
- [ ] TODO/FIXME格式规范
- [ ] 行内注释与代码有空格分隔

### 5.5 维护性检查
- [ ] 修改代码时更新相关注释
- [ ] 删除不再使用的注释代码
- [ ] 定期检查TODO/FIXME是否已完成

---
*本规范基于Doxygen文档生成工具，兼容C99及更高标准*
