## 验证流程

```
编码完成
    ↓
自动触发设计一致性验证
    ├─ 文档解析(提取功能、接口、模型)
    ├─ 代码扫描(glob/grep/read)
    ├─ 功能一致性检查
    ├─ 接口对应检查(含行号映射)
    ├─ 数据模型检查
    ├─ 代码存在性验证
    └─ 生成一致性报告
    ↓
验证结果判定
    ├─ 通过(≥90%) → 进入测试阶段
    ├─ 有条件通过(70-89%) → 修复后继续
    └─ 不通过(<70%) → 阻断流程,必须修复
```

## 验证内容

### 1. 功能一致性检查

检查设计文档中定义的所有功能是否在代码中有对应实现:

- 每个功能模块是否有代码实现
- 功能行为逻辑是否一致
- 输入输出是否匹配
- 异常处理是否完整

### 2. 代码与设计对应检查

验证设计要素与代码实现的对应关系:

- **模块对应**: 设计模块 → 代码目录/文件
- **接口对应**: 设计API → 代码路由实现(含行号)
- **数据模型对应**: 设计数据表 → 代码模型类
- **配置项对应**: 设计配置项 → 代码配置

### 3. 代码真实存在性检查

验证代码要素是否真实存在:

- 文件是否真实存在
- 函数/方法是否真实存在
- 是否为空实现(TODO/pass/空壳)
- 依赖模块是否已安装

### 4. 功能到代码行号映射

明确指出设计功能对应的代码位置:

```
设计功能: 用户注册
├─ 代码文件: src/auth/service.py
├─ 实现函数: register() (第18-42行)
├─ 接口路由: src/auth/routes.py (第15-28行)
└─ 数据模型: src/auth/models.py User类 (第15-45行)
```


## 配置说明

在 `.harness/config.yaml` 中增加一致性验证配置:

```yaml
harness:
  workflow:
    design_consistency_check: true  # 启用设计一致性验证
  
  design_consistency:
    enabled: true
    threshold: 90           # 一致率阈值(%)
    block_on_failure: true  # 验证失败是否阻断流程
    report_format: markdown # 报告格式
    
  quality:
    design_consistency: 90  # 设计一致性阈值
```

## 评级标准

### 功能一致性评级

| 评级 | 一致率 | 说明 |
|-----|-------|------|
| ✓ 一致 | ≥90% | 功能完全实现且符合设计 |
| ⚠ 基本一致 | 70-89% | 核心功能实现,存在非关键差异 |
| ✗ 不一致 | <70% | 功能缺失或严重偏离设计 |

### 综合评级规则

- **✓ 通过**: 所有维度一致率≥90%,无阻断性问题
- **⚠ 有条件通过**: 存在维度一致率70-89%,无阻断性问题
- **✗ 不通过**: 任一维度一致率<70% 或 存在阻断性问题

## 阻断性问题定义

以下问题会阻断流程继续:

1. **功能缺失**: 核心功能完全未实现
2. **接口无法访问**: API接口没有对应路由
3. **数据模型严重不匹配**: 关键字段缺失
4. **核心逻辑错误**: 实现与设计完全相反

## 最佳实践

### 1. 及时验证
- 编码完成后立即验证
- 每个功能模块完成后增量验证
- 提交前必须通过验证

### 2. 问题修复优先级
1. 阻断性问题: 立即修复
2. 一般问题: 优先修复
3. 改进建议: 后续处理

### 3. 文档同步
- 代码修改后更新设计文档
- 设计变更后重新验证
- 保持双向同步
